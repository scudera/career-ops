# OSS Evaluation — career-ops build vs buy (22/may/26)

> **Methodology**: 4 parallel research agents, ~60 min wall-clock, GitHub Code Search + `gh` CLI + WebFetch + npm/PyPI metadata + transient clones to `/tmp/oss-eval/`. Zero code changed in career-ops. All findings verified against actual repo contents (not just READMEs) where possible.

---

## TL;DR

- **BUY** (adopt OSS wholesale): **0** — no candidate replaces ≥2 career-ops providers with equal/better quality
- **PARTIAL** (import as dep): **0** — no narrow OSS lib justified as a dependency
- **STEAL** (port algorithms/data, no dep added): **6** — see Steal List below
- **SKIP**: ~20 — including JobSpy (TLS DPI dead-end, confirmed no v2), JobFunnel (archived Dec/25 by maintainer), OpenPostings (no LICENSE — legal blocker), jobber (dead since 2022), Feashliaa (strict subset of current coverage), generic Catho/Vagas scrapers (zero-star Selenium hobby projects)
- **Recommended migration scope**: **zero code migrated, zero deps added**. ~1–2 days of *additive* work porting STEAL intelligence into existing providers + adding 2–3 new providers (Gupy, RSS, Playwright). Schema expansion adds 5–7 fields to the `Job` contract.

### CP2 decision: **PROCEED as-is**, with high-value STEALs baked in

The provider-plugin architecture and ATS-API approach are **structurally correct** — independently validated by jobhive (Python, 47 ATS), ever-jobs (TS/NestJS, 160+ sources), and jakemercure28 (Node, 11 ATS) all converging on the same pattern. **JobFunnel's archival on Dec/2025** (with the maintainer citing "much more aggressive anti-automation") is the loudest signal in the niche: classic LinkedIn/Indeed scrapers are dying; per-ATS-API scrapers are alive. Career-ops is already in the right lane.

Career-ops's only structural gaps versus the OSS state of the art are:
1. **Schema-richness**: 4 fields (`title/url/company/location`) vs jobhive's 23 vs ever-jobs's 30
2. **Silent Workday 2K-cap bug** (`providers/workday.mjs:14` `MAX_OFFSET = 2000`)
3. **Zero BR pharma coverage** (no Gupy provider)
4. **No JD-detail extractor** (no JSON-LD parser exists in the codebase today)

All four are additive fixes, not architectural pivots. **Frente B v2 (CP2 parsers-from-scratch) is justified** — no OSS solves the JSON-LD semantic-normalization layer (jobLocationType + applicantLocationRequirements + multi-location + directApply). The "reinventing the wheel" hypothesis is **falsified**: the wheel does not exist.

## Critical findings (priority-ordered, action-ready)

### F1. Workday 2K-cap silent truncation **(BUG)**

`providers/workday.mjs:14` sets `MAX_OFFSET = 2000` and silently stops paging there. jobhive (`src/jobhive/scrapers/workday.py:90-103`) handles this by subdividing the search recursively along `jobFamilyGroup` → `timeType` → `workerSubType` facets when `total >= 2000`. **Impact**: Pfizer (~5K openings), J&J (~3K), Accenture (~60K), and every other large pharma Workday tenant lose ~60–95% of postings.

**Fix effort**: 2–3 hours port to Node.

### F2. Gupy has an unauthenticated public JSON API

Endpoint: `GET https://employability-portal.gupy.io/api/v1/jobs?jobName=...&workplaceTypes[]=remote|hybrid|on-site&limit=...&offset=...`. **No Bearer token, no API key.** Confirmed across 3 independent OSS scrapers (most recent: `lucasnunestrabalho99-sudo/telegram-vagas-gupy-bot`, 25 stars, 2026-05). Response shape: `{ data: [ { jobUrl, name, careerPageName, city, state, workplaceType, type, publishedDate, ... } ] }`. **This is not** the documented `api.gupy.io/v2/jobs` ATS-customer endpoint (which needs a Bearer token) — it's Gupy's own portal backend.

**Fix effort**: ~30 min for a complete `providers/gupy.mjs`. **Highest single-task ROI of the entire evaluation.** Closes ~60% of BR pharma posting gap.

### F3. JSON-LD JobPosting semantic parsing is a true OSS gap

Nine JSON-LD parsing candidates evaluated (extruct, web-auto-extractor, jsonld-extract, structured-data-testing-tool, metascraper, JobSpy, microdata-node, pickaxe, unhead schema-org). **None** parse `jobLocationType` / `applicantLocationRequirements` / multi-location / `directApply` into a typed object. The ecosystem splits cleanly into:

- (a) Generic JSON-LD passthrough (extruct, web-auto-extractor) — returns the raw `JobPosting` dict, leaves all coercion to the user
- (b) Wrong layer (JobSpy = list scraper; unhead = generator; metascraper = article metadata; pickaxe = GPL Java)

**Conclusion**: Building a 100–200 LOC custom cheerio-based JSON-LD parser is **correct**. STEAL extruct's HTML-comment-stripping + jstyleson-style lenient parse fallback + `@graph` recursion — that's ~30 LOC of robustness intelligence with no Python runtime needed.

### F4. Schema-richness, not provider count, is the competitive gap

| Library | Field count | Has `is_remote`? | Has `employment_type`? | Has `salary{}`? | Has `apply_url`? |
|---|---|---|---|---|---|
| **career-ops** (current) | **4** | ❌ | ❌ | ❌ | ❌ |
| jobhive (Python) | 23 | ✅ | ✅ | ✅ | ✅ |
| ever-jobs (TS) | ~30 | ✅ | ✅ | ✅ | ✅ |
| jakemercure28 (Node) | 7 | ❌ | ❌ | ❌ | ✅ |

Downstream code (`modes/oferta.md` scoring, work_mode filtering in `modes/_profile.md`, comp evaluation) is implicitly relying on inferences from the bare `location` string. **Adding 5–7 fields to the `Job` contract is the highest-leverage refactor** — pure additive change to the provider-plugin contract, no breaking interface for the existing 4 providers.

### F5. `applicantLocationRequirements` is a real moat

Across every OSS evaluated (ATS scrapers, JSON-LD parsers, pharma/RA aggregators, BR scrapers), **no library structurally surfaces geo eligibility** ("US-only", "EU-only", "ROW-eligible"). jobhive's `join_com.py:271` only checks `jobLocationType == "TELECOMMUTE"`. This is a genuine product moat — career-ops can ship a feature no public tool offers, with ~50 LOC of JSON-LD parsing in the eventual JD-detail extractor.

- License: MIT | Last commit: today (2026-05-23, HEAD: "Fix Phenom company entries #148") | 55 stars | 1 open issue

## Per-category verdicts

### Category 1 — ATS Scrapers — Verdict: **STEAL from jobhive, build remainder native**

The only OSS that covers what career-ops needs (Workday + Greenhouse + Lever + Ashby + Workable + Phenom + SmartRecruiters + work_mode) is **jobhive**, and it's Python. Adopting wholesale = full Python rewrite. Adopting via subprocess = adds runtime dep + breaks headless batch + complicates updates. The intelligence is mostly data tables (`_REMOTE_TYPE_PATTERNS`, `_TIMETYPE_TO_EMPLOYMENT_TYPE`) and small recursive algorithms (2K-cap subdivision). Port these into career-ops's existing Node providers. Add Workable/SmartRecruiters/Phenom as new `providers/*.mjs` files using jobhive's per-ATS Python files as reference.

### Category 2 — JSON-LD JobPosting Parsers — Verdict: **BUILD native, ~150 LOC**

No OSS solves the user's high-value field extraction (`jobLocationType`, `applicantLocationRequirements`, multi-location, `directApply`). The "reinventing the wheel" hypothesis is **falsified**. Steal extruct's robustness tricks; build everything else in cheerio. Test fixtures (10 real JDs from Greenhouse/Lever/Ashby/Workday/LinkedIn/Indeed/SmartRecruiters/Personio/Recruitee/BambooHR) are the real moat, not the parser.

### Category 3 — Self-hosted Job Aggregator Frameworks — Verdict: **No adoption, validate current architecture, steal schema**

JobSpy/JobFunnel/jobber are dead-ends. ever-jobs/jakemercure28 confirm career-ops's per-provider plugin pattern is correct. **Real finding**: career-ops's 4-field `Job` contract is the gap, not its coverage. Add `is_remote` / `employment_type` / `salary{}` / `posted_at` / `apply_url` / `description` to the contract (`_types.js`). All 4 existing providers + any new ones gain these fields incrementally.

### Category 4 — Pharma/RA/BR — Verdict: **Gupy is a 30-min win, RSS feeds are a 45-min win, EuroPharmaJobs is a dead-end**

Gupy's unauth `employability-portal.gupy.io/api/v1/jobs` endpoint is the single highest-ROI finding of this evaluation. Nature Careers + BioSpace RSS feeds are public and free — add a generic `providers/rss.mjs` and 2 lines in `portals.yml`. RAPS/PharmiWeb/EuroPharmaJobs need a Playwright-based provider because Cloudflare blocks ClaudeBot UA — career-ops already has Playwright wired for liveness, so extending it is ~2 hours. Skip Catho/Vagas specific scrapers (zero-star, Selenium-based, would rot fast).

### Phase 2 — Schema expansion (~4–6 hours)

4. **Add 5 more fields to Job contract**: `employmentType`, `compensation{}` (min/max/currency/period), `postedAt`, `applyUrl`, `descriptionHtml`. All optional, no breaking change. ### Phase 3 — New providers (~4–6 hours)

5. **Build `providers/workable.mjs`, `providers/smartrecruiters.mjs`, `providers/phenom.mjs`** using jobhive's per-ATS Python files as reference. ~1.5 hours each. Phenom adds CSRF handling complexity.
6. **Build `providers/rss.mjs`** generic feed reader. Add Nature Careers + BioSpace RSS to `portals.yml`. **45 min.**

### Phase 4 — JD-detail extractor (~6–8 hours, this is Frente B v2 / CP2 proper)

7. **Build `inspect-jd.mjs` (name flexible)**: cheerio-based JSON-LD JobPosting parser. Output the full `JobPostingDetail` type with `jobLocationType`, `applicantLocationRequirements` (array), `jobLocations` (multi), `directApply`, `baseSalary`, `validThrough`, `employmentType`. Steal extruct's robustness tricks. Test against 10 real-world JD fixtures. **6 hours.**
8. **Integrate into `pre-apply-check.mjs`** as an enrichment step before liveness — extract structured data once, reuse for both liveness and tracker enrichment. **1–2 hours.**

### Phase 5 — Playwright provider for protected sites (~2–4 hours, optional/deferred)

9. **Build `providers/playwright.mjs`** for Cloudflare-protected boards (RAPS, PharmiWeb, EuroPharmaJobs). Use existing career-ops Playwright wiring. Defer unless the user actually wants those sources — most pharma RA roles are findable via the other providers + RSS.

## What was rejected explicitly (anti-patterns dodged)

- ❌ "Just adopt JobSpy / it has 3.4k stars" — fails the user's TLS-DPI constraint; covers the wrong layer (boards, not ATSes); confirmed by user's own prior testing
- ❌ "Fork jobhive and add Workday improvements upstream" — fork = become maintainer of a fork = hidden cost; jobhive's Workday is already good
- ❌ "Use OpenPostings (249 stars)" — **no LICENSE file, all rights reserved by default**; legal blocker for any current or future commercial path
- ❌ "Migrate to NestJS / ever-jobs architecture" — NestJS is the wrong shape for a CLI; would discard pipeline.md / report numbering / liveness / CV-tailoring loop
- ❌ "Use pickaxe (Java) for JSON-LD" — GPL-3.0, flagged per user's license policy
- ❌ "Use schema-dts" — TypeScript types, not a parser; doesn't solve extraction
- ❌ "Just use cheerio + handwrite the parser" — this IS what we're recommending, but only because no OSS lib parses the semantic fields. Not a default; an evaluated outcome.
